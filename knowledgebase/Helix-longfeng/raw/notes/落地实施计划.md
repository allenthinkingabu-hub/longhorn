## 0. 文档说明

### 0.1 本计划的定位

这是一份**可执行**的落地计划，不是技术方案的复述。每个任务都带有责任人角色、人日估算、依赖关系、交付物清单和可量化的验收标准，团队成员看完应当能直接开工。

### 0.2 使用约定

- **任务编号**：`阶段.模块.序号`，例：`2.1.3` 表示阶段二·模板管理服务·第 3 个任务。
- **人日估算**：指完整投入一个资深工程师的工作日；新人按 1.5 倍折算。
- **角色缩写**：ARCH（架构师）、BE（Java 后端）、FE（前端/Web）、AND（Android）、IOS（iOS）、MP（小程序）、DEVOPS（运维/SRE）、QA（测试）、SEC（安全合规）、PM（项目经理）、DBA（数据库）。
- **依赖**：任务 A 依赖任务 B，表示 B 必须先完成（或至少进入可联调状态）A 才能启动。
- **交付物**：每项任务完成时应产生的可交付文件/服务/文档。
- **验收标准（DoD）**：Definition of Done，必须全部满足才能标记完成。

### 0.3 未决事项

以下内容在本计划中做了假设，进场前需业务方、法务、运维确认：

- 团队规模：按 1 名架构师 + 3 名后端 + 1 名 Android + 1 名 iOS + 1 名前端 + 1 名小程序 + 2 名运维 + 2 名测试 + 1 名 PM（共 12 人）估算。
- 硬件/云预算：核心部署在大连自建机房，青岛备用机房。如实际使用公有云，部分基础设施任务的工作量可缩减 30%。
- 合规：假设业务已具备 ICP 备案、《增值电信业务经营许可证》与基础用户隐私政策；如缺失，阶段 0 需增加 4-6 周法务流程。
- 推送通道账户：假设各厂商开发者账户在阶段 0 启动前已备案基础资质，仅需应用注册。

## 1. 项目概览

### 1.1 项目目标

建设一个覆盖 web / Android / iOS / 微信小程序的统一消息推送平台，支撑百万日活量级的实时、准时、定时推送，满足模板化扩展、多语言、个性化、离线补发的核心需求。

### 1.2 关键指标（SLO）

| 指标 | 目标值 | 备注 |
|---|---|---|
| 推送请求 QPS | ≥10 万 | 集群峰值 |
| 接口平均响应时间 | ≤200 ms | P95 |
| 任务调度延迟 | ≤100 ms | 准时率 ≥99.9% |
| 系统可用性 | ≥99.99% | 月度统计 |
| 消息到达率 | ≥99% | 按通道统计，厂商通道目标 ≥99.9% |
| 离线消息补发成功率 | ≥99% | |
| 个性化变量填充耗时 | ≤50 ms | 单消息 |
| RPO / RTO | ≤5 min / ≤30 min | 异地切换 |

### 1.3 非目标（本期不做）

- 富媒体推送（音视频内嵌）
- AI 智能推荐引擎（仅接入已有推荐服务）
- 国际化海外部署（仅保留接口扩展位）
- 私有化输出（仅 SaaS 模式）

### 1.4 关键依赖与约束

- 需要完成小米、华为、OPPO、vivo、苹果（APNs）、微信小程序等开发者平台的应用注册与资质审核。
- 需要法务介入，完成《个人信息保护法》合规评估与推送授权文本起草。
- 大连机房需提前 4 周完成网络、电力、机柜的交付验收。
